This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 项目性质

这是一个 Obsidian 笔记仓库（Vault），位于 `/home/shiwu/Documents/Note`。对笔记内容的任何操作，必须优先使用 Obsidian 相关 Skill。

## 必须使用的 Skill

操作笔记时，根据任务类型选择对应 Skill：

- **obsidian-markdown**: 创建或编辑笔记内容（.md 文件），包括 wikilinks、callout、frontmatter、标签、嵌入语法等 Obsidian 特有语法。
- **obsidian-cli**: 搜索笔记、查看 vault 信息、管理任务/属性、开发调试插件等命令行操作。
- **json-canvas**: 创建或编辑白板文件（.canvas），用于流程图、思维导图。
- **obsidian-bases**: 创建或编辑数据库文件（.base），用于表格视图、卡片视图、筛选、公式。
- **defuddle**: 从网页中提取干净的 Markdown 内容，用于保存网页资料到笔记。

## 笔记操作规则

### 基本原则

1. **不得改变笔记原意**：编辑已有笔记时，只做用户明确要求的修改，不擅自改写、润色或扩展内容。
2. **不得产生歧义**：新增或修改的内容必须表述清晰，避免模糊用语。技术术语保持准确。
3. **保持笔记风格一致**：使用中文撰写；遵循下文所有格式规范。

### 双向链接规范（Wikilink）

双向链接是 Obsidian 的核心功能，必须充分利用：

1. **任何笔记都必须考虑潜在关联**：阅读或编辑笔记时，主动识别其中提到的技术、工具、概念是否已有对应笔记，如有则添加 wikilink。
2. **链接必须是双向的**：如果笔记 A 链接到笔记 B，那么笔记 B 中也应当添加回到笔记 A 的链接。单向链接是残缺的。
3. **链接粒度**：可以链接到整个笔记（`[[Linux/KVM]]`），也可以链接到特定标题（`[[Linux/KVM#嵌套虚拟化]]`），根据上下文选择最精确的目标。
4. **链接显示文本**：当链接文本在上下文中需要调整时，使用管道符指定显示文本（`[[Linux/KVM|KVM 内核虚拟机]]`）。
5. **任务清单链接**：每日任务中涉及的具体事项（如"配置 KVM/QEMU"、"学习 Rust"），必须链接到对应的知识笔记。
6. **新建笔记后必须添加反向链接**：为新主题创建笔记后，检查所有已有笔记中是否有提及该主题的地方，并补上链接。

### 文本格式规范（Markdown + Obsidian）

充分利用 Markdown 和 Obsidian 扩展语法增强可读性，避免大段纯文字。

#### 行内格式

| 语法 | 写法 | 适用场景 |
|------|------|---------|
| **加粗** | `**text**` | 首次出现的关键术语或概念；操作步骤中的关键动作（如"**安装** KVM 软件包"）；需要用户特别注意的语句 |
| *斜体* | `*text*` | 引用外部术语、书名；轻度强调；英文缩写展开说明（如"*Central Processing Unit*"） |
| ***粗斜体*** | `***text***` | 同时需要加粗和斜体的极端强调 |
| ==高亮== | `==text==` | 关键参数或配置值；容易出错的步骤；快捷键表格中的重要快捷键；需要一眼看到的数值 |
| `行内代码` | `` `code` `` | 正文中提及的命令名、文件名、参数名、包名、路径、端口号、版本号 |
| ~~删除线~~ | `~~text~~` | 已废弃的内容、旧版本配置、不再有效的命令 |
| 外部链接 | `[显示文本](url)` | 引用外部网页、官方文档 |
| 脚注引用 | `[^1]` | 对正文内容需要补充说明但不打断阅读流时使用 |

#### 块级元素

| 语法 | 写法 | 适用场景 |
|------|------|---------|
| **标题** | `# H1` ~ `###### H6` | `#` 为笔记主标题（仅一个），`##` 为主要章节，`###` 为子章节，层级不得跳跃 |
| **无序列表** | `- ` 或 `* ` 或 `+ ` | 并列关系的条目、功能列表 |
| **有序列表** | `1. ` `2. ` `3. ` | 有先后顺序的步骤、排名、流程 |
| **任务列表** | `- [ ] ` / `- [x] ` | 待办事项、任务清单、检查清单 |
| **代码块** | ` ```language ` ... ` ``` ` | 所有命令、代码、配置文件内容，必须带语言标识（`shell`、`bash`、`toml`、`yaml`、`json`、`python`、`rust` 等） |
| **表格** | `| col1 | col2 |` | 多组同类数据的对比、快捷键表、配置参数表 |
| **引用块** | `> text` | 引用外部内容、摘录原文 |
| **水平分隔线** | `---` 或 `***` 或 `___` | 主题切换、章节间的视觉分隔 |
| **脚注定义** | `[^1]: 内容` | 与脚注引用配合，放在文末 |

#### Obsidian 专属语法

| 语法 | 写法 | 适用场景 |
|------|------|---------|
| **Wikilink** | `[[路径/文件名]]` | 链接到其他笔记（双向链接的核心），详见双向链接规范 |
| **带别名的 Wikilink** | `[[路径/文件\|显示文本]]` | 链接文本与目标文件名不同时使用 |
| **标题级链接** | `[[路径/文件#标题]]` | 精确链接到笔记中某个章节 |
| **标题级别名链接** | `[[路径/文件#标题\|显示文本]]` | 精确链接到章节，同时自定义显示文本 |
| **嵌入笔记** | `![[路径/文件名]]` | 将另一篇笔记的完整内容嵌入当前笔记 |
| **嵌入图片** | `![[路径/图片名.png]]` | 嵌入附件中的图片 |
| **嵌入音频** | `![[路径/音频.mp3]]` | 嵌入附件中的音频 |
| **嵌入 PDF** | `![[路径/文件.pdf]]` | 嵌入 PDF 文件 |
| **嵌入章节** | `![[路径/文件#标题]]` | 仅嵌入笔记中的某个章节 |
| **Callout** | `> [!type] 标题` | 提示信息块（见 Callout 使用规范） |
| **可折叠 Callout** | `> [!type]- 标题` | 默认折叠的提示块，内容较长或不常用时使用 |
| **标签** | `#tag` 或 `#层级/子标签` | 正文中的标签（与 frontmatter 中的 tags 互补） |
| **评论/注释** | `%% 内容 %%` | 仅在编辑模式可见的隐藏注释，不影响渲染输出 |
| **Mermaid 图表** | ` ```mermaid ` ... ` ``` ` | 流程图、时序图、类图、思维导图等 |
| **Math 公式** | `$inline$` 或 `$$block$$` | LaTeX 数学公式（行内和块级） |

### Callout 使用规范

使用 Obsidian callout 标注不同类型的提示信息，不得将所有提示都用纯文本表达：

| Callout 类型 | 适用场景 | 示例 |
|-------------|---------|------|
| `> [!note]` | 补充说明、概念解释 | `> [!note] KVM 是 Linux 内核模块，QEMU 是用户态模拟器` |
| `> [!tip]` | 实用技巧、快捷操作 | `> [!tip] 使用 --help 可查看所有子命令` |
| `> [!notice]` | 需要留意的信息 | `> [!notice] 默认配置不含桌面环境` |
| `> [!warning]` | 可能导致问题的操作 | `> [!warning] 修改此配置后需重启 libvirtd` |
| `> [!important]` | 关键信息，必须关注 | `> [!important] TPM 2.0 是 Windows 11 的硬性要求` |
| `> [!info]` | 背景知识、原理说明 | `> [!info] 嵌套虚拟化依赖 CPU 的 VMX/SVM 特性` |

原则：每个笔记中的关键提示、注意事项、技巧至少使用一个 callout，长流程笔记（如安装教程）中每个重要步骤都应有对应的 callout。

### Frontmatter（属性）规范

所有笔记必须包含 YAML frontmatter（`---` 包裹的元数据），确保一致性：

1. **必选字段**：
   - `date`: 创建日期（格式 `YYYY-MM-DD`）
   - `tags`: 至少 2 个标签，使用 `/` 表示层级（如 `linux/kvm/virtualization`）
2. **可选字段**：
   - `title`: 笔记标题（与文件名不同或需要明确指定时使用）
3. **任务清单笔记额外要求**：
   - `tags` 中必须包含 `tasks/daily`（每日任务）或 `tasks`（其他任务）

### 标题层次规范

1. 每篇笔记必须有且仅有一个 H1（`#`）标题。
2. 使用 H2（`##`）划分主要章节，H3（`###`）划分子章节。
3. 标题层级不得跳跃（不能从 H2 直接跳到 H4）。
4. 任务清单笔记必须包含 `#` 主标题（如 `# 任务清单`）。

### 笔记编辑后检查清单

编辑或创建笔记后，确认以下事项：

- [ ] 有完整的 frontmatter（date、tags）
- [ ] 有且仅有一个 H1 标题
- [ ] 标题层次清晰，无跳跃
- [ ] 大段纯文字已转为列表或表格
- [ ] 关键术语/参数使用了加粗或高亮
- [ ] 所有命令/路径/包名都在代码块或行内代码中
- [ ] 适当的提示使用了 callout（每个笔记至少一个）
- [ ] 与相关笔记建立了双向 wikilink
- [ ] 任务清单中的事项链接到了对应知识笔记
- [ ] 已扫描 vault 全局，确认不存在遗漏的反向链接

### 笔记优化工作流

每次优化笔记时，必须遵循以下流程：

```
拉取最新 → 优化笔记 → 提交 → 移动书签 → 推送
```

具体步骤：

1. **拉取最新**：`jj git fetch`，确保本地与远程同步。
2. **检查冲突**：若远程有新提交，`jj rebase -s <本地提交> -d <远程提交>` 将本地工作变基到远程之上。
3. **优化笔记**：严格遵循以下优化规则（详见下方"优化规则"）。
4. **确认变更**：`jj st` 查看所有变更，排除不应提交的文件（如 `.obsidian/workspace.json`）。
5. **提交**：`jj commit -m "描述"`，提交信息使用中文，简洁描述做了什么更改。
6. **移动书签**：`jj bookmark move main --to <提交ID>`，每次提交后必须将 `main` 书签移动到新提交。
7. **推送**：`jj git push`。

> [!important] 先拉取再工作
> 在开始任何笔记修改之前，必须 `jj git fetch`。这避免与远程（如 Mobile Sync）产生冲突。

### 优化规则

优化笔记时，必须严格执行以下规则：

#### 1. 使用 Obsidian Skill

编辑或创建 `.md` 文件时，**必须**通过 `obsidian-markdown` Skill 进行，确保 wikilink、callout、frontmatter 等 Obsidian 特有语法的正确性。

#### 2. 先扫描再优化

**在修改任何笔记之前**，先扫描 vault 中所有笔记，识别以下优化机会：

- 哪些纯文本可以转为表格（如对比信息、参数列表）
- 哪些大段文字可以拆分为列表
- 哪些关键术语/参数尚未加粗或高亮
- 哪些步骤缺少 callout 提示
- 哪些已有笔记可以建立双向 wikilink

> [!tip] 批量优化
> 如果发现多处相同问题，一次性批量处理，避免反复提交。

#### 3. 充分发挥 Markdown 语法

根据内容语义选择最适合的格式，**不得出现大段纯文字**：

| 内容类型 | 应使用 |
|---------|--------|
| 步骤流程 | 有序列表 |
| 并列关系 | 无序列表 |
| 对比信息、参数表 | 表格 |
| 命令/代码/路径/包名 | 代码块或行内代码 |
| 关键术语、首次出现的概念 | **加粗** |
| 关键参数、配置值、易错点 | ==高亮== |
| 警告、技巧、注意事项 | Callout |

#### 4. 充分发挥双向链接

- 每个笔记必须与所有相关笔记建立 `[[wikilink]]`
- 链接必须是双向的：A 链 B，则 B 必须链 A
- 任务清单中的事项必须链接到对应知识笔记
- 新建笔记后必须扫描已有笔记，补上反向链接

### 注意事项

- **不要写 "Co-Authored-By: Claude"**：这是 git 习惯，jj 提交不需要。
- 提交信息使用中文，简洁描述做了什么更改。
- 书签 `main` 跟踪主分支，`jj git push` 默认推送当前书签。
- **`jj commit` 之后必须移动书签**：提交后 `main` 书签不会自动前进，不移动书签则 `jj git push` 不会推送任何内容。
- `.obsidian/workspace.json` 已在 `.gitignore` 中，不应提交。
